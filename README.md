# Avito Seller - Frontend Test Assignment

Веб-приложение для личного кабинета продавца на Avito с интеграцией AI-ассистента.

## Запуск проекта (интеграция с Ollama)

## Запуск с Docker

- Для скорости и точности модели лучше разворачивать приложение без Docker, см. раздел [Локальный запуск](#локальный-запуск-без-docker).

### Development сборка
```bash
docker-compose up --build
```

### Production сборка
```bash
docker-compose -f docker-compose.prod.yml up --build
```

Примечание:
- При первом запуске Docker скачает образ `ollama/ollama` (несколько GB).
- При первом запуске будет автоматически скачана лёгкая модель `llama3.2:3b` в Docker volume `ollama_data`.
- Запросы могут идти медленно из-за ограничений RAM в контейнере (можно увеличить в настройках Docker).

Дополнительно:
- Если нужно качество выше, можно вручную поставить `llama3`:
  - В файле `docker-compose.yml` изменить значение entrypoint на `ollama serve & sleep 2; ollama show llama3 >/dev/null 2>&1 || ollama pull llama3; wait`

Очистка места от Ollama:
```bash
docker-compose down
docker-compose -f docker-compose.prod.yml down
docker volume rm avito-test_ollama_data
```

```bash
docker-compose down
```

## Локальный запуск (без Docker)

### Установка Ollama
```bash
# Скачайте и установите Ollama с официального сайта
# https://ollama.ai/download

# Загрузка модели
ollama pull llama3

# Запуск сервера
ollama serve
```

### Запуск проекта
```bash
# Терминал 1: Backend
cd server
npm start

# Терминал 2: Frontend
cd client
npm run dev

# Терминал 3: Ollama
ollama serve
```

## Принятые решения и обоснования

### Backend изменения

**Добавление id в ответ GET /items**
- **Решение**: Добавлено поле `id` в каждый элемент ответа
- **Обоснование**: Для списка объявлений нужен стабильный идентификатор для навигации на страницу объявления. Без ID невозможно реализовать переход к детальной странице объявления.

**Добавление сортировки по price**
- **Решение**: Реализована сортировка по полю `price` в GET /items
- **Обоснование**: Контракт должен соответствовать UI и требованиям задания. Пользовательский интерфейс включает сортировку по цене, что требует соответствующей поддержки на бэкенде.

**Разрешение изменения category в PUT /items/:id**
- **Решение**: Добавлено поле `category` в схему валидации для PUT запроса
- **Обоснование**: Форма редактирования по ТЗ допускает смену категории, что логично с точки зрения пользовательского опыта. Продавец может ошибиться при выборе категории или захотеть её изменить.

### Подход к разработке

**Минимальные изменения backend**
- **Решение**: Изменения сведены к минимально необходимым без перестройки архитектуры сервера
- **Обоснование**: Сохранить дух тестового задания и не усложнять код. Добавлена только необходимая функциональность для корректной работы frontend.

**Фокус на UI/UX**
- **Решение**: Приоритет отдан пользовательскому опыту и соответствию современным практикам
- **Обоснование**: Цель тестового задания — продемонстрировать умение создавать удобный интерфейс с учётом технических ограничений.
